# 项目目录规范

## 适用对象

本规范适用于“江苏省高校图工委教改课题申报 + 后续落地实施”项目库。目标是避免申报材料、调研资料、方案设计、实施文档、治理留痕相互混放，导致后期难以迭代和复用。

## 目录角色

| 目录 | 角色 |
| --- | --- |
| `00-总览` | 项目入口，只放导航、全局状态、关键链接 |
| `01-规范` | 当前有效规则，如目录规范、命名规则、文稿标准 |
| `02-管理` | 任务计划、里程碑、风险、推进节奏、协同事项 |
| `09-治理` | 审计结论、决策记录、版本说明、变更依据 |
| `10-项目工作` | 唯一主工作目录，内部按模块组织申报、方案、实施、调研、评估 |
| `60-资产` | 可复用模板、图示、术语表、通用素材 |
| `90-存档` | 被替代版本、历史稿、废弃方案 |
| `98-附件` | PDF、截图、原始非正文材料 |
| `99-设置` | 辅助脚本、模板、工具说明 |

## 入口规则

- `00-总览` 只承担总入口职责，不替代 `10-项目工作` 主工作驾驶舱。
- `00-总览` 应优先链接稳定入口页，如 `10-项目工作`、`01-规范`、`09-治理`、`98-附件`，不应直接承担深层正文分发。
- 若确有必要展示深层正文，应单列“当前重点文档”一类的次级区域，避免总览页退化为文件清单。
- `10-项目工作` 是当前 proposal 工作区唯一正式主工作入口；辅助导航页如被替代，应转入 `90-存档`。

## 工作目录模块

`10-项目工作` 内部统一采用模块化编号：

| 模块 | 角色 |
| --- | --- |
| `10-课题申报` | 申报书、汇总表口径、证明材料清单、答辩口径 |
| `20-方案设计` | 解决方案、业务蓝图、技术路径、阶段建设方案 |
| `30-落地实施` | 立项后的建设设计、接口清单、测试、部署、运维材料 |
| `40-调研案例` | 外部高校案例、政策材料、标杆实践、文献摘要 |
| `50-评估论证` | 可行性、预算、风险、伦理、推广价值、绩效评估 |

## 核心边界

- `10-项目工作/10-课题申报` 放“对外报送文本”，不放长期管理台账。
- `10-项目工作/20-方案设计` 放“你准备做什么、为什么这么做”，不放会议纪要。
- `10-项目工作/30-落地实施` 放“怎么做、谁来做、如何验收”，在申报阶段可先留空或只放预研。
- `10-项目工作/50-评估论证` 单独成层，避免可行性分析散落在申报书和方案文稿里。
- `09-治理` 只记录结论性事项，不替代日常推进。
- 课题标题、范围边界、技术路线、经费口径等重大调整，应先登记到 `09-治理/项目变更台账.md`，再回写正文。

## 关键节点审计规则

- 本项目采用“关键节点必须评审”的项目级审计规则，不允许只靠主观推进进入下一阶段。
- 每到一个关键节点，必须生成一份 `review` 评估报告，并给出建设性的改善意见。
- 每份关键节点评估都必须同时包含：
  - 当前结论与是否通过该节点
  - 主要问题清单，按 `P1 / P2 / P3` 分级
  - 建设性改善意见，不只指出问题，还要说明如何改
  - 量化评分
  - 雷达图或等价的多维度可视化评分图
- 关键节点评估结果统一归口到 `10-项目工作/50-评估论证`。

当前最少应设置以下关键节点：

1. `选题与主叙事收敛后`
2. `方案边界与一期范围冻结前`
3. `申报书送审稿形成时`
4. `报送定稿前`
5. `立项后实施方案启动前`

建议默认评估维度如下：

- 目标契合度
- 教改问题与价值
- 创新性与辨识度
- 可实施性与路径完整度
- 研究基础与团队准备度
- 资源保障与协同条件
- 合规性与报送完整度

评分要求如下：

- 默认采用 `100` 分制，可按维度权重计算总分
- 雷达图维度数应与评分表一致，避免图表与正文口径不一致
- 关键节点的 `svg` 评分图或推进图应显式标注其对应的报告名称，避免图表脱离上下文单独传播后失去归属
- 若关键节点评估未清除主要 `P1` 风险，不应视为通过该节点
- 未产出评估报告和量化图表的节点，不视为完成

## 命名规则

- Markdown 正文优先使用 `中文语义名.md`。
- Folder Note 文件属于例外：为满足 Obsidian `folder-notes` 插件要求，可与目录同名，并保留模块编号，如 `10-项目工作.md`、`10-课题申报.md`。
- 稳定基线文件默认不强制带日期，如规范、台账、当前采用的方案、模块 MOC。
- 一次性过程文件应显式带日期，日期格式统一使用 `yy.MMdd`，如 `项目管理目录审计报告-26.0423.md`。
- 一次性过程文件包括：分析报告、调研快照、审计意见、评审快照、重构说明、决策快照、临时导出说明、阶段草稿快照。
- 草案文件应显式暴露生命周期状态，如 `草案`、`填报稿`、`审计意见`、`要点摘录`。
- 草案文件显式标识状态，如 `课题申报书草案.md`、`应用创新方案纲要.md`。
- 当同类草稿或过程文件可能并存多个版本时，应同时显式暴露 `对象 + 状态 + 日期`，不要只写 `草案.md`、`审计意见.md`。
- 不使用含糊命名，如 `方案.md`、`材料.md`、`文稿最终版.md`。

## 当前结构为什么这样设计

本项目有两个并行目标：

- 目标一：完成省级教改课题申报。
- 目标二：让图书馆智能体创新真正落地。

如果把申报、方案、调研、评估直接平铺成多个顶层目录，入口会变多，工作体验会发散；如果完全混在一起，又会失去边界。当前采用“一个主工作目录 + 五个模块”的结构，就是为了解决入口统一与职责清晰这两个问题。
